rescue

Rescue data from failing storage

ddrescue-style block imaging, plus a file carver that keeps your videos whole instead of splitting them into fragments.

View on GitHub Download a release

Why another carver?

Tools like PhotoRec guess where a video file ends by sniffing its content, and give up as soon as the data stops looking like video — the result is one clip split into dozens of tiny fragments. rescue carve instead parses the container structure (the ftyp/moov/mdat boxes in MP4/MOV/HEIC, the EBML elements in MKV/WebM, the declared sizes in RIFF/ASF) and follows the declared length to the file's exact end. A contiguous video comes out as one file, byte-exact, every time.

Files genuinely fragmented on disk (interleaved writes on FAT/exFAT) can't be fixed by any contiguous carver — that needs a filesystem-aware pass, which is on the roadmap.

How it works

  1. 1. Image the failing device

    rescue image copies a dying block device into a sparse image file, tracked in a GNU ddrescue-compatible mapfile: easy data first, then it narrows down bad sectors in passes (copy → trim → scrape → retry). Interrupt it any time — re-running the same command resumes exactly where it left off. Linux only, since it needs direct block-device access.

  2. 2. Carve files out of the image

    rescue carve scans the image for file signatures and extracts photos, videos and documents — structure-aware where the format allows it, so extraction is exact rather than guessed. Pass --map and it skips regions that were never rescued, and flags any extracted file that overlaps a bad sector as damaged in the report. Works on Linux, macOS and Windows.

Install

Grab a binary from the releases page, or build it yourself with Cargo:

git clone https://github.com/kuboon/file-rescue
cd file-rescue
cargo build --release
# binary at target/release/rescue

Usage

Image a failing device

# never opens the source for writing; refuses to write
# the image onto the source device itself
rescue image /dev/sdb card.img card.map --retry-passes 2

# interrupted? just run the same command again to resume
rescue image /dev/sdb card.img card.map --retry-passes 2

Check the damage

rescue map show card.map
rescue map regions card.map --status bad-sector

Carve files out

rescue carve card.img recovered/ --map card.map
cat recovered/report.txt

Try the whole flow without real broken hardware:

cargo run --example mkdemo -- demo.bin
cargo build --features fault-injection
target/debug/rescue image demo.bin out.img out.map \
    --simulate-bad 0x140000+0x2000,0xD08000+0x400@2 --retry-passes 1
target/debug/rescue carve out.img recovered/ --map out.map

Supported formats

FormatsHow the length is found
MP4 / MOV / HEIC / AVIF / 3GP (ISO-BMFF)box-structure walk exact
MKV / WebM — the usual home of VP9/AV1EBML element sizes exact
AVI (incl. >2 GiB OpenDML) / WAV / WebP (RIFF)declared RIFF size exact
WMV / WMA (ASF)object sizes exact
PNGchunk walk exact, damaged files carved partially
GIFblock-structure walk exact
BMPdeclared file size exact
TIFF and TIFF-based RAW (CR2; typically NEF/ARW/DNG)IFD walk to furthest strip/tile data
JPEGmarker validation + entropy-aware EOI scan
MPEG-PS (.mpg — DVD recorders)pack/PES walk by start codes
MPEG-TS (.ts) / AVCHD (.m2ts)188/192-byte packet sync run
PDFlast %%EOF (incremental updates handled)

AV1 is a codec, not a container — AV1 video is recovered via its container (MP4/WebM/MKV), AV1 stills via AVIF. Adding a format is one file plus one registry line; see the README.
